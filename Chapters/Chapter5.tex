\chapter {Pong as a Testbed}
\label{chap:pong}
Pong provides a controlled setting in which to compare the two paradigms developed in the previous two chapters. The environment is complex enough to require learning visual representations from raw pixels, yet simple enough that both \acrshort{dqn} and \acrshort{a2c} can be trained to play on a single GPU reasonably. This chapter formalizes Pong as a Markov decision process, describes the preprocessing pipeline that transforms raw frames into a suitable state representation, and presents the implementation and training results for each method.

\section{Environment Description and Preprocessing Pipeline}
\label{sec:env_preprocessing}
Pong is a game containing two paddles and one ball in a court. The agent controls a paddle and the other one is controlled by a built-in opponent. A round ends when the ball passes one paddle. A player wins when 21 points are scored.

The mapping of Pong to the finite \acrshort{mdp} in \eqref{eq:finite_mdp} is straightforward. The state space $S$ consists of the raw pixel image of 210x160x3 values returned by the Gymnasium \acrshort{ale} wrappers at each timestep. This embeds the ball position, velocity, paddle positions and the score which are not directly accessible and must be learned by the agent. The action space $A$ is discrete and consist of 6 values namely NOOP (does nothing), FIRE (does nothing), RIGHT (go up), LEFT (go down), RIGHTFIRE (go up), LEFTFIRE (go down). The transition dynamics $p$ is based on deterministic physics and is defined in the \acrshort{ale} environment. The agent has no access to $p$ which must be learned through interaction making the learning approach model-free. The reward signal $R$ is sparse where the agent receives a +1 when it scores, a -1 when the opponent scores and 0 otherwise. This sparsity makes Pong a meaningful testbed, testing whether an algorithm can propagate reward backward through long sequences of actions. The discount factor $\gamma$ is set to 0.99, a standard choice for Atari environments where reward-producing actions may precede the actual reward by many timesteps.

The \acrshort{mdp} formalism assumes that the environment follows the Markov property of encapsulating all that is required to act optimally. The observations should allow the states to be distinguishable from one another. In the case of Pong, a single frame is not enough to learn the speed and direction of objects like the ball and the paddle. This type of environment is called a partially observable \acrshort{mdp} \citep{kaelblingPlanningActingPartially1998}. To overcome this, \citet[p.~534]{mnihHumanlevelControlDeep2015} stacks 4 most recent frames together which gives the agent a short history, approximating true \acrshort{mdp}. Moreover, to reduce the demand on computation and memory, the input is rescaled to give an image of size 84x84x4 pixels.

The preprocessing pipeline \citep[p.~534]{mnihHumanlevelControlDeep2015} additionally includes taking the component-wise maximum over pixel values across two consecutive frames to remove flickering. To reduce the input size by a factor of three, only the grayscale channel is kept. 4 frames in the input are skipped so that the agent always notices a change.

Both \acrshort{dqn} and \acrshort{a2c} use the exact same preprocessing so that any performance difference in Section~\ref{sec:dqn_results} and Section~\ref{sec:a2c_results} is attributable to the algorithm and not the input representation.

\section{DQN Implementation and Results}
\label{sec:dqn_results}

\subsection{Implementation}

The network architecture for the \acrshort{dqn} is in listing $\hyperref[code:dqn_core_model]{model.py}$. The network consists of three convolutional layers. The filter count, kernel size, and stride of the layers are 32@8x8/s4, 64@4x4/s2, 64@3x3/s1 and all use ReLU activations. The last layer is a 512-unit fully connected layer ending in one linear unit per action and produces the Q-Value estimates, and not probabilities, for all actions simultaneously.

The training procedure in listing $\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ uses the network [line 95], the agent [line 116], the loss calculation [line 44] and the optimizer [line 119] and ends when the agent reaches a reward of 19. Six engineering and training speedup code optimizations that are not present in \citet{mnihHumanlevelControlDeep2015} have been applied as described in Section~\ref{sec:dqn-optimizations} which allows the training to complete within 28 minutes on a system with specifications listed in Table~\ref{tab:system-configuration}.

The agent in listing $\hyperref[code:dqn_core_dqn_agent]{dqn\_agent.py}$ interacts with the environment using a $\epsilon$-greedy policy [lines 96-103] derived from the Q-network [line 101]. The $\epsilon$-schedule is in $\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ [line 40] and uses the number of parallel environments in it's calculation. The exploration is loaded early in the training and then gradually shifted towards exploitation as the Q-value estimates improve (\ref{sec:mdpframework}). The transitions in the \acrshort{mdp} are stored in an experience replay buffer with 100000 transition capacity $\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ [lines 34, 115]. At each training step, a minibatch of size 64 [$\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ line 179] is sampled uniformly at random [$\hyperref[code:dqn_core_dqn_agent]{dqn\_agent.py}$ line 59] which breaks temporal correlations as described in Section~\ref{sec:stabilization}. The loss is computed against the target network $Q(s, a; \theta_i^{\,\text{--}})$ [$\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ line 64] and continuously synchronized with the online network using Polyak averaging [$\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ lines 189-190]. The optimizer used is Adam with a learning rate of 0.0001 [$\hyperref[code:dqn_dqn_pong]{dqn\_pong.py}$ line 119].

A $\hyperref[code:dqn_core_wrappers]{wrappers.py}$ extends required Gymnasium API to stack 4 observations together to approximate true \acrshort{mdp} [lines 44-70]. It also rearranges observations to the dimensions required by Pytorch [lines 27-41]. The creation of the game environment is also wrapped to add preprocessing defined in Section~\ref{sec:env_preprocessing}.

After the training is complete, the $\hyperref[code:dqn_dqn_play]{dqn\_play.py}$ is used to play a game of pong which creates a recorded video of the game on disk.
